kCFNull

NSNull 的用法 Mattt Thompson 说过是将空值存到数组或者字典

NSMutableDictionary *mutableDictionary = [NSMutableDictionary dictionary];
mutableDictionary[@"someKey"] = [NSNull null]; // Sets value of NSNull singleton for `someKey`
NSLog(@"Keys: %@", [mutableDictionary allKeys]); // @[@"someKey"]

kCFNull 这个是什么呢，看定义可以知道

typedef const struct CF_BRIDGED_TYPE(NSNull) __CFNull * CFNullRef;
const CFNullRef kCFNull;    // the singleton null instance

看个栗子

NSNull *null1 = (id)kCFNull;
NSNull *null2 = [NSNull null];

输出信息

(NSNull *) null1 = 0x0000000107b10af0
(NSNull *) null2 = 0x0000000107b10af0

说明 (id)kCFNull 就是 [NSNull null]

How To Record Your App With QuickTime

Posted on 2015-11-12

之前为 Github 上的开源项目录制了一个 Demo 的 Gif，因为有些硬盘洁癖，所以没有下载其他录屏工具，使用的是 Mac 自带的 QuickTime，之后将录制的视频转成 Gif，可是隔段时间以后就忘了如何录制了，在此记录下过程。

将程序在你的 iOS 设备上运行，之后打开 QuickTime 进行以下操作：

  • 点击 “File”
  • 点击 “New Movie Recording”
  • 在出现的窗口中点击录制按钮旁边的三角箭头，选择你的 iOS 设备，开始录制吧

视频录制后使用 GIFBrewery 工具生成 Gif，该工具可以设置捕捉帧数及图片颜色数量，基本需求都能满足。

- (void)actionSheet:(UIActionSheet *)actionSheet clickedButtonAtIndex:(NSInteger)buttonIndex {
    SEL selectors[] = {
        @selector(insertRow),
        @selector(insertSection),
        @selector(deleteSection)
    };

    if (buttonIndex < sizeof(selectors) / sizeof(SEL)) {

        imp(self, selectors[buttonIndex]);

NSURLSession 工作模式

NSURLSessionConfiguration 分为三种工作模式
default: 可以使用磁盘缓存、身份认证、Cookie
ephemeral(及时模式): 数据只能保存到内存，无法存储到磁盘。
background: 可以在后台执行上传或下载任务

后进先出

后进先出通过 NSOperation 的依赖实现

if (wself.executionOrder == SDWebImageDownloaderLIFOExecutionOrder) {
        // Emulate LIFO execution order by systematically adding new operations as last operation's dependency
        [wself.lastAddedOperation addDependency:operation];
        wself.lastAddedOperation = operation;
}

GCD barrier

dispatch_barrier_async 比较常见，那么 dispatch_barrier_sync 又是做什么的，区别是什么呢？看文档可知 dispatch_barrier_async 不用执行完内部方法即可返回，dispatch_barrier_sync 需要等到执行完内部方法才会返回

        // ...
    }

Mobile Client Development Notice

Posted on 2014-01-15

这是一篇关于移动客户端开发中容易出现的Bug及问题以及产品方面的大杂烩。

写代码前应当注意的问题

  1. 进度条、提示框的处理。在需要超过 2s 的地方添加进度条或等待提示框。出错时要使用提示框提示用户能看得懂的提示语句。
  2. 删除、放弃等操作需要添加确认提示。
  3. 当设计稿只给定了一种文字样式，这种样式正好文字不是很长时，需要确认文字过多、过长时的显示方式。文字过多时是省略(使用…)还是截取，是否支持多行显示。
  4. 图片的显示方式，图片显示时是要进行裁剪还是等比缩放。
  5. 小屏手机上的显示效果。例如出现键盘显示时，如果设计给出的效果在小屏手机上出现键盘遮挡控件时，需要确认如何处理。
  6. 列表点击后是否需要设置选中色。
  7. 需要使用数据库存放数据时，图片、文件、 二进制数据不要存放到数据库中。

提交测试前应确认的问题

  1. 网络异常时程序是否出现异常。
  2. 带输入框的地方没有输入或输入过长时的处理，是否使用了合适的键盘，纯数字或者文本？能否正确地触发键盘的弹出和收起。
  3. 如果调用系统 API 获取了时间，当本地时间发生变动时是否会造成影响。
  4. 按钮是否支持连续点击。
  5. 列表为空时展示成何种样式，网络出现问题时是否需要支持重新加载。
  6. Tab 切换是否正常刷新。

产品相关方面应当考虑的问题

  1. 想不清楚一个功能点时宁可不做，或者突发奇想觉得某个功能很好玩时，先要三思。
  2. 不要期待你的客户帮你想如何改进，他不是你的产品经理。
  3. 如果大家都在为老板做产品，“哎，大家都觉得这里很别扭，大家都觉得这样很难看，但是改吧，老板说要这个功能，老板说要做成这样”，这个时候就要想想是否能试图改变这种状况，否则考虑考虑离开这家公司吧。
  4. 同一个页面有多个入口，同一个功能有多种展现方式？你让用户选择，不如你为用户选择好一个最佳的方式。
  5. 保持产品主干清晰，一个产品应该主要关注一件事情，如果分支太多，或者喧宾夺主，你以为你留存了用户，但是那样的用户大多情况下不是你想要的用户，如果你拉投资需要算人头除外。
